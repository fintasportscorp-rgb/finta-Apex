Le plan de frappe est défini par cette construction géométrique, identique entre le Python d'origine et le TypeScript :
                                                             
  ---
  Les 3 points impliqués                                                                                                                              
  - hip : hanche du côté dominant (ex. hanche droite pour un droitier)                                                                               
  - other_hip : hanche opposée                                                                                                                        
  - wrist : poignet dominant                                                                                                                          
                                                                                                                                                    
  Les 2 vecteurs                                                                                                                                    
  hip_vector = hip → other_hip   (ligne des hanches)                                                                                                    arm_vector  = hip → wrist      (bras du jeu, depuis la hanche)
                                                                                                                                                        Projection XZ (vue du dessus)                                   

  Les deux vecteurs sont projetés sur le plan horizontal XZ — on ignore Y (hauteur). La coordonnée Z de MediaPipe est la profondeur par rapport à la
  caméra, donc XZ = vue aérienne. C'est la clé : on mesure l'ouverture de stance vue d'en haut, pas la hauteur du bras.

  Calcul de l'angle

  angle_brut = acos(dot(hip2d, arm2d) / (|hip2d| × |arm2d|))
  hitting_plane = angle_brut - 90°

  Le -90° : quand le bras est perpendiculaire à la ligne des hanches (stance neutre), l'angle entre les vecteurs est exactement 90°. On soustrait 90
  pour que le neutre = 0°.

  Convention de signe via produit croisé

  cross = hipVecX × armVecZ - hipVecZ × armVecX

  - Droitier : cross > 0 → bras en arrière → fermer → négatif
  - Gaucher : cross < 0 → même sens → négatif

  Résultat :
  - + = stance ouverte (poignet en avant de la ligne des hanches, vers le filet)
  - − = stance fermée (poignet derrière)
  - ≈0° = stance neutre (bras perpendiculaire aux hanches)

  ---
  Le TypeScript (hittingPlane.ts:44-66) est une traduction ligne par ligne de calculate_hitting_plane_angle() dans le Python
  (BH_hitting_plane2.py:149-194). La seule différence : en Python on passe les tableaux numpy directement, en TS on extrait .x/.z des landmarks
  MediaPipe.

  Diagnosis                                                                                                                     
                                                                                                                                
  What's Working                                                                                                                
                                                                                                                                  - findRawPatterns / aggregatePeaks: cycle detection across multiple reps is solid                                             
  - findAllPatterns / findRiseOnset: peak detection + onset within mean series works                                              - Gantt + coach text generation: structural logic is correct    

  Root Causes (3 distinct bugs)

  Bug 1 — colorIndex assigned before sort (kineticChain.ts:330 vs 356)
  colorIndex is set to the loop index i (script order), then result.sort() reorders the array. After the sort, m.colorIndex
  still points to the pre-sort position. Colors in the Gantt (COLORS[m.colorIndex]) are randomly misaligned with the row they
  appear in.

  Bug 2 — Sort order is sequence-dependent (kineticChain.ts:356-360)
  result.sort((a, b) => (a.refPeakT ?? a.peakT) - (b.refPeakT ?? b.peakT))
  Without a reference, sorts by each sequence's own peakT. Sequence A may peak knee_flex at 0.2 and shoulder at 0.4; Sequence B
  may peak shoulder at 0.1 and knee at 0.5. The Gantt row order flips between sequences → impossible to compare.

  Bug 3 — computeRepOnsets and computeChain use independent colorIndex (line 298 vs 330)
  computeRepOnsets assigns colorIndex by script position; computeChain also assigns by script position but then re-sorts. If a
  measure is at row 3 in computeRepOnsets but row 1 in the sorted Gantt, the pattern area overlay on line charts has the wrong
  color for the Gantt bar.

  What's Missing

  - Individual rep visibility: aggregatePeaks collapses all reps to one number. No way to see if rep 1 had hip_shift before
  knee_flex while rep 2 had them reversed.
  - Movement signature: no canonical ordering defined per sport/script — no way to say "forehand always starts with knee_flex"

  Without a reference, sort by peakT causes the order to vary per-sequence. The fix: only sort when refInstances is provided
  (sort by refPeakT, which is stable). Without a ref, keep script order.

  1b. Reassign colorIndex after sort
  After the sort, forEach((m, i) => m.colorIndex = i) so colors always = sorted row position.

  1c. computeRepOnsets accepts chain and derives colorIndex from it
  export function computeRepOnsets(instances, chain: MeasureChain[]): RepOnset[]
